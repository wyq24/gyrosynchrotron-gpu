# gyrosynchrotron-gpu — Project Handoff
Date: 2026-04-05

## 1. Project goal

This project is converting part of the gyrosynchrotron package into a practical GPU-capable workflow.

Current priority is **engineering optimization**, not paper outputs.

The project philosophy is:

- keep the validated baseline stable
- optimize in narrow, additive steps
- do not broaden physics/workflow scope unless explicitly justified
- preserve compatibility with the current Python workflow

---

## 2. Current validated scope

The currently validated GPU scope is still intentionally narrow:

- Python only
- analytical `PLW + ISO`
- `Nz = 1`
- approximate GS only
- batched workflow first
- current legacy-compatible Python wrapper remains intact

Out of scope for now:

- exact GS
- array DF
- transport beyond the already mirrored single-voxel RL conversion
- IDL
- broader API redesign
- external `pygsfit` wrapper changes

## 5. FP32 status

FP32 is **not completely unusable**, but it is **not acceptable as a reliable optional mode yet**.

### What has been fixed already
Two major catastrophic FP32 failure classes were removed:

1. invalid-wave sentinel conversion problem
2. dangerous normalization / overflow problem (`norm_a`-related)

After those fixes:
- `inf / nan` failures were eliminated
- catastrophic zero-collapse was removed
- FP32 became finite and benchmarkable

### What still remains
The remaining dominant FP32 error source is **not** a simple local expression anymore.

Current diagnosis:
- remaining above-floor FP32 outliers come from the corrected-root path:
  - `FindMu0`
  - `H1`
  - corrected `mu0`
  - `lnq2`
- downstream variables like `x`, `q_pre`, and `z` then amplify those differences
- wrapper outliers track local emissivity outliers, so this is **not primarily a wrapper problem**

Representative remaining FP32 errors before branch pause:

- local real-sweep:
  - `jx` worst max-rel: `9.318e-02`
  - `jo` worst max-rel: `2.432e-01`
- local stress-sweep:
  - `jx` worst max-rel: `3.239e-03`
  - `jo` worst max-rel: `1.178e-02`
- wrapper real-sweep:
  - left worst max-rel: `1.668e-01`, max-abs `1.263e-03`
  - right worst max-rel: `1.158e-01`, max-abs `1.092e-01`
- wrapper stress-sweep:
  - left worst max-rel: `7.342e-03`, max-abs `1.492e-02`
  - right worst max-rel: `3.239e-03`, max-abs `1.167e-01`

### FP32 performance
FP32 is much faster, but still not trustworthy enough.

Representative FP32 wrapper speedup vs CUDA FP64:

- batch 8: ~`1.97x`
- batch 32: ~`1.86x`
- batch 128: ~`2.26x`
- batch 512: ~`2.61x`

Representative FP32 wrapper speedup vs legacy CPU wrapper:

- batch 8: ~`4.94x`
- batch 32: ~`14.09x`
- batch 128: ~`30.93x`
- batch 512: ~`49.40x`

### Final FP32 branch decision
Current decision:
- FP32 would require broader mixed precision in the corrected-root / `lnq2` path
- that is beyond current highest-ROI optimization
- therefore `opt/fp32-salvage` is paused

## 6. Important engineering conclusions

## 6.1 Python binding is not the main end-to-end bottleneck
Measured wrapper/backend gap was dominated by **Python-side RL/output postprocessing**, not by:

- ctypes binding cost
- Python/native call boundary
- host-to-device transfer
- device-to-host transfer

This means:
- there is no strong reason to replace the current Python calling convention right now
- the correct optimization direction is additive native-side postprocessing fusion

---

## 6.2 Legacy compatibility should be preserved
Current preferred strategy:

- keep the current legacy-compatible Python wrapper
- keep the current native local `j/k` batch path
- add fast paths only in an additive way
- do not redesign public APIs unless later evidence clearly demands it

---

## 6.3 Best current optimization direction
Highest-ROI current optimization path:
- continue on `opt/fused-native-rl`
- integrate the fused native RL fast path into the normal supported Python batch workflow
- automatically use the fast path only for the validated narrow case
- fall back to the old Python RL path for unsupported cases

This preserves compatibility while improving real end-to-end speed.
